# Runbook — upgrading a brain to v0.9.0

Created: 2026-07-05

v0.9.0 is a **security-hardening** release. Two of its changes fail *closed*,
so a mis-configured brain can refuse to start (by design) — read the whole
runbook before upgrading. See [`RELEASE_NOTES.md`](../RELEASE_NOTES.md) for the
full change list.

Do the brains **one at a time**. Upgrade a low-stakes brain first and the
**public demo brain last**, so a mistake surfaces on a brain nobody depends on
before it can touch the public surface.

---

## 0. Before you touch anything (per brain)

- Know how this brain was deployed. Most fleet brains are provisioned by
  **rsync**, not `git clone`, and may have a stale or absent `.git` — in that
  case `git pull` does nothing and you redeploy by re-running your rsync step
  (see `SERVERS.md` / your provisioner). If the brain *is* a real git checkout,
  `git pull` is fine. **Confirm which before assuming `git pull` updated the code.**
- Have the brain's `.env` open. You will edit it (see step 2).

## 1. Get the v0.9.0 code onto the brain

For a git-checkout brain:

```bash
cd /path/to/brain          # e.g. the BRAIN_HOST_DIR for this brain
sudo chown -R "$USER:$USER" .   # api container runs as root; avoids a silent-fail pull
git pull origin main
git rev-parse --short HEAD  # confirm it matches origin/main
```

For an rsync-provisioned brain: re-run your rsync deploy so the working tree
matches `main` at v0.9.0. Verify the tree carries this release (e.g. `VERSION`
reads `0.9.0`).

## 2. Configure the new fail-closed / hardening settings in `.env`

These are the v0.9.0 changes that need operator action. Edit each brain's `.env`
by hand (do **not** commit secrets):

1. **`TRUST_PROXY_HEADERS` — set to `true` for EVERY brain behind Caddy (or any
   reverse proxy).**

   ```
   TRUST_PROXY_HEADERS=true
   ```

   Why it matters: with the v0.9.0 default (`false`), the brain uses the direct
   transport peer for public-surface rate limiting. Behind Caddy the transport
   peer is *Caddy's* IP, so **every visitor collapses into one rate-limit bucket**
   and legitimate users get throttled as one. Set it `true` **only** when a
   trusted proxy is really in front (Caddy appends the real client IP as the last
   hop). A brain reachable directly on its port must leave it `false`.

2. **Postgres password must be strong.** In non-dev (`BRAIN_ENV != dev`) the API
   now refuses to start if `POSTGRES_PASSWORD` is empty or `postgres`. If this
   brain still has the default, set a strong one first:

   ```
   POSTGRES_PASSWORD=$(openssl rand -hex 32)   # put the value in .env
   ```

   (Changing an existing DB password also means updating the running Postgres
   role / volume — if the brain has real data, rotate deliberately, don't just
   edit `.env` and expect the old volume to accept the new password.)

## 3. The public **demo** brain only — extra steps (do this brain LAST)

The demo brain serves `/v1/public/chat` to the internet, so it also needs:

- **`TURNSTILE_SECRET_KEY`** set (Cloudflare Turnstile) so bot loops can't drain
  the endpoint. Add the site key to the public-chat front end as well.
- **`PUBLIC_CHAT_DAILY_MAX`** explicitly set (default is now 2000; keep or tune —
  do **not** set `0`/unlimited on the public brain).
- **`PUBLIC_CHAT_MODEL` stays local** (an Ollama model). Do **not** point the
  public surface at a paid API model without a spend cap + Turnstile (tracked in
  issue #2). Local model = abuse costs CPU, not cash.

## 4. Rebuild and restart

```bash
docker compose up -d --build
```

`--build` is **mandatory**, not optional. The image bakes the Python deps at
build time; only a `--build` picks up the current `requirements.txt`, which is
what makes the clean `pip-audit` result true of the *running* brain rather than
just of a fresh install. A plain `docker compose up -d` would restart the old
image and silently keep stale deps.

Watch it come up:

```bash
docker compose logs -f api    # confirm no "Startup refused: ..." message
curl -fsS http://127.0.0.1:8010/health
```

If the api container refuses to start, the log line tells you which guard
tripped (weak Postgres password, missing `BRAIN_IDENTITY_SECRET`/`BRAIN_API_KEY`,
etc.) — fix the `.env` value and re-run step 4.

## 5. Manual smoke test — one real chat round-trip

A green build is not proof the brain answers. For the **public demo brain**,
after step 4, do one real round-trip against the live public endpoint and
confirm tokens actually stream from the real model:

```bash
curl -N -X POST https://<demo-brain-domain>/api/chat \
  -H 'Content-Type: application/json' \
  -d '{"message":"hello — are you streaming?","history":[]}'
```

Expected: a `text/event-stream` response with `data: {"token": "..."}` frames
arriving incrementally, ending in `data: {"done": true}` — generated by the
brain's real `PUBLIC_CHAT_MODEL`, not a stub. If you instead get a single JSON
error (rate limit, 5xx, or `Upstream unreachable`), the public path is not
healthy — investigate before considering the brain launched.

For a non-public brain, do the equivalent authenticated `/chat/rag` round-trip
in the console.

## 6. Post-upgrade checklist (per brain)

- [ ] Code is at v0.9.0 (`VERSION` = 0.9.0 / HEAD matches `origin/main`).
- [ ] `TRUST_PROXY_HEADERS=true` if behind Caddy.
- [ ] Postgres password is non-default; api container started (no "Startup refused").
- [ ] (demo only) Turnstile set, `PUBLIC_CHAT_DAILY_MAX` set, model is local.
- [ ] `docker compose up -d --build` run (not a plain `up`).
- [ ] Smoke test passed — real SSE tokens streamed from the real model.

Order recap: low-stakes brains first, **public demo brain last**.
