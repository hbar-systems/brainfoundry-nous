# Kernel Governance — PROPOSE → CONFIRM
*BrainFoundryOS / NodeOS*

---

## What it is

The kernel governance system is the **boundary between intent and execution** in a brain node. No mutation — to system state, users, config, or data — can happen in a single step. Every action command must first be proposed, receive a confirmation token, and then be re-submitted with that token before anything executes.

This is not a UX pattern. It is a structural guarantee: **an AI agent acting autonomously inside the brain cannot unilaterally cause irreversible changes.**

---

## The problem it solves

A brain node is an AI system with the ability to act — on your behalf, with your identity, against your data. That capability is the point. It is also the attack surface.

Without a governance boundary:
- A prompt injection in a chat message could trigger a destructive command
- A compromised loop permit could mutate the brain's identity or wipe its memory
- An agent running autonomously has no structural check on its own actions

With PROPOSE → CONFIRM:
- The agent can propose anything — but proposing does nothing
- Execution requires a second, explicit call with a token that was issued only for that specific proposed action
- The token is time-limited and single-use — a forged or replayed token fails
- Every proposal and confirmation is logged to an append-only audit trail

The confirmation step is harder to fake than the proposal step. An injected prompt might get a proposal through. It cannot get a confirmation through without the token that was just issued — which means a human (or an explicitly authorized kernel process) had to receive it and choose to use it.

## How it works

### Command structure

```
<verb> <noun> [parameters] [--confirm TOKEN]
```

Verbs: `help`, `list`, `show`, `create`, `update`, `delete`, `execute`

Read-only verbs (`help`, `list`, `show`) return results immediately — no confirmation needed.

Mutating verbs (`create`, `update`, `delete`, `execute`) enter the PROPOSE → CONFIRM flow.

---

### Step 1 — PROPOSE

Submit a mutating command without a token:

```
create user name=john role=admin
```

Response includes five structured sections:

```
INTENT:        The interpreted intent of the command
PLAN:          Proposed plan of action
RISKS:         Assessment of potential risks
OUTPUT:        (empty until confirmed)
CONFIRMATION:
  status: PROPOSED
  token: CONFIRM-a1b2c3d4
  instructions: re-run the command with --confirm CONFIRM-a1b2c3d4
```

Nothing has happened. The brain has only declared what it would do.

---

### Step 2 — CONFIRM

Re-submit the same command with the token:

```
create user name=john role=admin --confirm CONFIRM-a1b2c3d4
```

Token is verified. If valid:

```
CONFIRMATION:
  status: CONFIRMED
  Token verified successfully.
```

If invalid or expired:

```
CONFIRMATION:
  status: DENIED
```

---

### Bad token demonstration

The Kernel page in the brain UI includes a **CONFIRM (bad token)** button specifically to demonstrate the denial path — showing that the system rejects unverified tokens rather than silently accepting them.

Similarly, submitting a command that doesn't exist (e.g. `__does_not_exist__`) returns `KERNEL_UNKNOWN_COMMAND` — demonstrating that even unknown commands go through the boundary and fail explicitly rather than silently.

## Interfaces

### Web UI — Kernel tab

Available at the `/kernel` tab in the brain console. Fields:

| Field | Description |
|-------|-------------|
| CLIENT_ID | Identifying caller (e.g. `demo`) |
| COMMAND | The verb-noun command |
| PAYLOAD.TEXT | Optional payload |
| CONFIRM_TOKEN | Auto-filled after PROPOSE; submit empty for proposal |

Endpoint: `POST /api/bf/v1/brain/command`

Requires `HBAR_BRAIN_API_KEY` — passed as `Authorization: Bearer <key>` header.

### CLI — Local mode

```bash
python console.py --interactive
python console.py create user name=john role=admin
python console.py create user name=john role=admin --confirm CONFIRM-a1b2c3d4
```

Audit logs written locally to:
- `command_audit.jsonl` — all commands and responses
- `proposals.jsonl` — proposals and confirmation tokens

### CLI — Remote mode

```bash
python console.py --remote https://brain.hbar.systems --api-key YOUR_KEY create user name=john role=admin
```

Audit log held on server. Token generated by server, not client.

---

## Audit trail

All commands and confirmations are logged to append-only `.jsonl` files. Existing entries are **never modified, truncated, or deleted**. This is a hard invariant — the audit trail is the proof of what was proposed, when, and whether it was confirmed or denied.

---

## Current implementation state (v0)

As of v0: the token flow is fully implemented (PROPOSE issues a token, CONFIRM verifies it), but **execution does not yet happen** — confirmation returns `CONFIRMED (NO EXECUTION IN V0)`. The governance boundary is live; the execution layer behind it is the next build milestone.

This is intentional. The governance model is proven before the execution capability is wired in.

---

## Future: Declarative governance policy in brain.toml

Currently policy is hardcoded in the registry. The next evolution is declarative policy in `brain.toml`:

```toml
[governance]
mode                    = "propose-confirm"
confirm_threshold       = "owner"
rag_trust_threshold     = "owner"
destructive_delay_s     = 30        # wait before executing destructive commands
federation_requires_did = true      # peer introductions need DID signature
auto_confirm_read_only  = true      # skip confirm loop for safe reads
```

The kernel scans those rules on every action. That's governance as constitutional infrastructure — not a UI pattern, but a formal policy engine that any node operator can configure without touching code.

---

## Why this matters for federation

When two brains talk to each other, the receiving brain needs to know that an inbound action was authorized by its owner — not by a rogue node, an injected message, or a relay attack. The PROPOSE → CONFIRM model, combined with the loop permit system and node identity layer, is what makes brain-to-brain action trustworthy rather than just technically possible.

The kernel is not just a safety feature for solo use. It is the foundation for sovereign federation.
